# mockoon-gen 公开文章优化记录

## 目标与读者

目标是面向掘金和 Notion 的公开技术文章，而不是 skill 的内部操作手册。读者关心的是：为什么需要这套方案、它如何解决真实项目问题、自己需要做哪些业务决策，以及生成的文件如何投入联调。

## 背景模块的写法

背景必须以 `## 背景` 明确标注，并使用第一人称、具体项目经历推进，不要直接从工具定义或结论切入。推荐叙事顺序：

1. 项目中遇到的阻塞：后端未就绪、联调环境不稳定、页面仍需开发和验证。
2. 已尝试方案：在代码中写死响应、接入 MSW 等。
3. 每种方案失败的机制和影响面：代码污染、多人协作的开关冲突；自研微前端下公共配置、请求初始化或 worker 注册会影响多个子应用，并可能进入审核。
4. 最终技术决策：使用独立运行的 Mockoon；为了保持原始 API 域名，再以 Whistle 做本机流量转发。
5. 引出 skill：重复维护接口契约、Mockoon 配置、Whistle 规则的成本，才是创建 `mockoon-gen` 的直接动机。

背景不应只罗列“代码侵入性强、影响面广”这类结论。每个判断要给出项目中的具体落点和因果关系，才能有足够信息密度。

## 工具比较的写法

- MSW 只介绍一次：先肯定其适用场景，再说明本项目为何不采用。
- 不把 MSW 描述为能力不足；问题是项目的微前端架构和治理边界使其接入成本变高。
- “为什么需要 Whistle，而非只启动 Mockoon”属于决策背景和原理，应在开头说明。核心事实是 Mockoon 不会改变浏览器请求目的地，Whistle 才能在不修改 `baseURL` 的情况下把原始 API host 转向本机服务。
- 同时写清 Whistle 的代价：代理链路、HTTPS 证书信任，以及只应匹配必要 host/路径。

## skill 使用流程的写法

以使用者视角组织流程，不以内部 CLI 实现组织：

```text
普通接口文档 → OpenAPI → 人工 review → 用户决策 → 生成产物 → GUI / CLI / Whistle 联调
```

其中“普通接口文档生成 OpenAPI”必须是显式的第一步，不能只在输入说明中一笔带过。应说明普通文档要补齐路径、方法、请求参数、成功/失败响应、字段类型和示例；生成器不能替代接口 review。

文章只展开需要用户决策的内容：

- 是否确认 OpenAPI；
- 是否开启随机空值模式；
- 列表场景是否开启、数量是否足够覆盖 UI；
- 语义明确的字符串字段应使用何种 Faker 映射；
- Mockoon 端口；
- 是否需要 Whistle、目标 API host、规则组名称；
- Whistle 选择 `json` 还是 `cjs`。

不要展示 `mockoon-gen` 的底层 CLI 命令、参数、路径约束、隐藏目录限制、强制覆盖策略等内部实现细节。它们对公开读者的任务成功没有直接帮助，也会把文章写成内部说明书。

## 产物与运行说明的写法

先一次性介绍所有产物及各自职责，再集中说明如何消费它们：

1. `openapi.yaml`：review 后的契约；
2. `mock-artifact.json`：审查与生成中间产物；
3. `mockoon.json`：Mockoon GUI 和 CLI 的输入；
4. `whistle.json` / `whistle.cjs`：Whistle 的输入。

之后按运行工具分节：Mockoon GUI、Mockoon CLI、Whistle。不要在流程中途插入 GUI 或 CLI 操作，也不要在每一步重复解释同一产物。

## “Why mockoon-gen skill”模块的写法

这一模块不应只是“功能清单”，而要回答读者已经接受 Mockoon + Whistle 方案后自然会问的问题：**为什么还需要一个生成 skill？** 标题可直接使用读者问题，例如 `## why mockoon-gen skill`，让它承接背景中的“手工维护配置麻烦”。

推荐的内容顺序如下：

1. **先给结果。** 一句话说明 skill 根据接口文档和项目配置生成 Mockoon 数据配置与 Whistle 规则；导入两个文件后即可 Mock 对应接口。
2. **补足默认能力。** 在同一段中交代它默认覆盖成功、空数据、错误、列表，以及可选的随机空数据场景，避免读者只把它理解成格式转换器。
3. **用流程图解释跨格式转换。** 从“各团队形态不同的普通接口文档”开始，经过标准 OpenAPI 和 `mock-artifact.json`，再分流到 `mockoon.json` 与 `whistle.json` / `whistle.cjs`。流程图应表达数据流和中间审查层，而不是重复命令步骤。
4. **用产物表收束。** 每一行只回答“这个文件是什么、给谁用”，不提前展开 GUI、CLI、Whistle 的具体操作。

这类模块的关键信息密度来自“输入不统一 → 中间契约统一 → 两类运行产物”的因果链。比起逐项解释内部字段或校验规则，更能说明 skill 的存在价值。

措辞上应保持通用：主体是 `mockoon-gen` skill，不绑定某个 AI 客户端；`mock-artifact.json` 只说明为场景、审查项、语义映射和输出配置服务，不把内部实现细节扩展成使用说明。

## 参考文献的收敛与编号

参考文献不是“调研时看过的链接清单”，而是正文中仍在使用的外部事实来源。发布前执行两次核对：

1. 从正文提取所有 `[[n]](URL)`，确认每个编号在参考文献中恰有一条对应来源。
2. 从参考文献反查正文；没有被任何正文引用的条目应删除。

编号按正文首次出现的顺序连续排列。删除条目后必须同步重排正文和列表，不能留下跳号。参考条目要与正文链接指向同一页面：例如正文使用 Mockoon 的 `about` 教程，就不要在列表中写成另一个“数据文件”页面。

只保留能够支撑当前版本内容的最小来源集。普通安装入口或项目主页链接不等同于需要论证的外部事实，可以直接使用链接，不必编号为参考文献。对已在改稿中删除的 MSW 描述、旧版 GUI 说明或未使用的 CLI 细节，应同时删除相应参考文献，避免让读者误以为文章仍依赖这些论据。

## 信息密度检查

发布前逐段检查：

- 这一段是否给出了新事实、决策条件、操作结果或明确取舍？若没有，删除或并入前后段。
- 是否把同一个结论换词重复，例如多次解释 MSW 的工程侵入性？保留机制最清楚的一次。
- 是否把 skill、Codex 或某个客户端当作主体？公开文章应以 `mockoon-gen` 这个通用 skill 为主体，只在安装/触发处提及兼容的客户端。
- 是否把内部实现细节误当成使用说明？读者只需要知道输入、决策、产物和验证方式。

## 发布节奏

先在本地 Markdown 完成所有模块的审阅和修改，再同步到 Notion。避免在结构尚未定稿时反复全量更新页面；Notion 发布版本应与最终可发布 Markdown 一致。

# writing-style-yzin 公开技术文章增强设计

## 目标

在保留现有通用技术写作能力的前提下，为面向外部读者的公开技术文章增加可执行的叙事、决策和信息取舍规则，避免将文章写成内部操作手册。

## 范围

- 保留中文写作、结构与深度、论证与事实边界、引用和语气等既有规则。
- 在 `SKILL.md` 中新增一个仅面向公开技术文章的章节。
- 不增加引用清理流程、发布同步流程或内部技术文档分流规则。
- 不增加脚本、参考文件或其他资源。

## 设计

新增章节通过“仅在任务明确为对外发布的技术文章时适用”限定作用范围，并包含以下约束：

1. 以 `## 背景` 说明具体项目阻塞、既有尝试、失败机制与实际影响，再引出方案。
2. 比较工具时先说明替代方案的适用场景，再结合当前项目的架构或治理边界说明取舍。
3. 将技术决策的原理、收益与代价置于操作步骤之前。
4. 按使用者任务组织为“输入 → 人工 review → 用户决策 → 产物 → 联调”，不按内部实现或命令顺序组织。
5. 仅保留读者完成任务所需的输入、决策、产物、消费方式和验证信息，排除不影响任务成功的内部实现细节。
6. 集中介绍全部产物的职责和消费者，再按运行工具说明消费方式。
7. 说明 skill 的价值时，以“输入不统一 → 统一契约与审查层 → 运行产物”的因果链为主，避免功能清单式介绍。
8. 要求逐段提供新事实、决策条件、操作结果或明确取舍；合并或删除换词重复的内容。

## 验证

- 检查新增章节是否清楚标明适用场景，且不影响其他技术文档任务。
- 检查规则是否覆盖已确认的九项问题，且未引入排除项。
- 运行 skill 的基础格式校验，确认 YAML frontmatter 与目录结构有效。
